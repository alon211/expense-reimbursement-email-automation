# 系统架构总览

**最后更新**: 2026-03-11

本文档提供报销邮件自动化服务的高层架构视图，包括项目概述、核心特性、技术栈和系统架构图。

---

## 项目概述

### 什么是报销邮件自动化服务？

一个基于 Python 的邮件自动化处理服务，用于从 IMAP 邮箱中拉取匹配规则的报销邮件，提取正文和附件，并通过钉钉推送通知。

### 核心价值

- **自动化**: 定时拉取邮件，无需人工干预
- **智能化**: 基于规则的邮件匹配，智能去重
- **结构化**: 提取并分类存储邮件内容
- **可扩展**: 模块化设计，易于添加新功能

---

## 核心特性

### 1. 基于规则的邮件匹配
- JSON 配置规则（rules/parse_rules.json）
- 多维度匹配（发件人、主题、正文）
- 规则启用/禁用开关
- 时间范围配置

### 2. 智能去重机制
- 基于邮件发送时间的唯一标识
- 预检查去重（快速匹配）
- 文件存在性验证
- 三层防护机制

### 3. 邮件内容提取
- 邮件正文 HTML 保存
- 附件提取和解码
- 压缩文件自动解压（zip、rar、7z、tar.gz）
- 结构化数据导出（JSON）
- 按规则分类存储

### 4. 数据库管理
- SQLite 数据库
- 邮件记录表（extracted_emails）
- 提取历史表（extraction_history）
- 索引优化
- 数据库迁移支持

### 5. 钉钉消息推送
- Webhook 消息发送
- 签名验证（HMAC-SHA256）
- 推送开关控制
- 汇总通知（多邮件合并）

### 6. Docker 容器化部署
- Dockerfile 镜像构建
- docker-compose.yml 编排配置
- 环境变量注入
- 数据持久化挂载

## 数据流图

### 邮件处理流程

```
用户邮箱
    │
    │ 1. IMAP 连接
    ▼
邮件搜索（SINCE {date}）
    │
    │ 2. 逐个解析
    ▼
规则匹配（RuleLoader）
    │
    ├─→ 匹配失败 → 跳过
    │
    └─→ 匹配成功
         │
         │ 3. 去重检查
         ▼
    mail_date in existing_mail_dates?
         │
         ├─→ 是 → 跳过（已提取）
         │
         └─→ 否
              │
              │ 4. 内容提取
              ▼
        EmailExtractor
              │
              ├─→ 保存邮件正文 HTML → bodies/{rule_id}/
              ├─→ 提取附件 → attachments/{rule_id}/
              ├─→ 解压压缩文件 → attachments/{rule_id}/{name}_extracted.{ext}/
              └─→ 导出 JSON → extracted/{rule_id}/
              │
              │ 5. 数据库记录
              ▼
        DatabaseManager
              │
              ├─→ 添加/更新 extracted_emails 表
              └─→ 添加 extraction_history 表
              │
              │ 6. 消息推送
              ▼
        DingtalkNotifier
              │
              └─→ 推送汇总通知 → 钉钉机器人
```

## 关键设计决策

### 1. 为什么使用 SQLite？
- **轻量**: 无需独立数据库服务器
- **跨平台**: Windows/Linux 兼容
- **零配置**: 无需安装和配置
- **足够**: 满足当前数据量需求

### 2. 为什么使用 mail_date 作为唯一标识？
- **可靠**: 邮件发送时间由邮件服务器生成
- **稳定**: 同一封邮件的发送时间始终相同
- **跨会话**: 即使程序重启，发送时间也不变
- **易用**: 方便快速去重检查

### 3. 为什么使用 JSON 规则配置？
- **易读**: 人类可读可编辑
- **灵活**: 易于添加新规则
- **无依赖**: 无需额外配置文件解析库
- **版本控制**: Git 友好

### 4. 为什么使用 Docker？
- **环境隔离**: 避免依赖冲突
- **部署简单**: 一键启动
- **可移植**: 跨平台运行
- **可扩展**: 易于水平扩展

## 扩展点

### 1. 新增规则类型
在 `rules/parse_rules.json` 中添加新规则即可

### 2. 新增推送渠道
参考 `dingtalk_notifier.py` 创建新的推送模块

### 3. 新增数据源
参考 `email_fetcher.py` 创建新的邮件拉取模块（如 Exchange）

### 4. 新增文件处理
参考 `archive_utils.py` 创建新的文件处理模块
